# ASR -> AI纠错稿设计

## 背景

`course-ai` 当前会把 ASR 分段结果直接写入 `transcripts` 表，后续的文稿面板、AI 笔记、章节、总结、测验、思维导图、RAG 检索和字幕导出都复用这份文本。

用户希望把 `ASR -> AI纠错稿` 打通为新的默认主线，但保持前台体验简洁：

- 只有在已配置可用大模型时才自动执行 AI 纠错
- 前台默认直接显示纠错后的最终稿，不特别提示
- 原始 ASR 结果只做后台备份
- AI 纠错失败时，整条处理流程仍然算成功，保留原始 ASR 文稿
- 纠错应尽量保留原时间戳分段，只做中度润色：修正识别错误、病句、口语赘词和标点，但尽量贴近原话

这次设计只覆盖 `ASR -> AI纠错稿` 主线，不包含后续的讲义 PDF 和截图提问实现。

## 目标

1. 在不打断现有使用流程的前提下，自动把可用的 ASR 文稿提升为更干净的正式文稿。
2. 保持 `transcripts` 作为全仓库统一的“当前正式文稿”来源，避免为每个消费方单独加兼容逻辑。
3. 在后台保留原始 ASR 快照，给后续回退、重跑和能力扩展留出空间。
4. 把失败影响控制在最小范围：AI 纠错失败不能拖垮整条处理链。

## 非目标

- 不做原始稿 / 纠错稿双栏对照界面
- 不做用户人工审核纠错差异
- 不新增单独的“转写纠错”模型配置入口
- 不在本次实现里改造讲义 PDF、截图提问或新的导出格式
- 不把 AI 纠错拆成新的显式顶层处理阶段

## 用户体验

### 成功路径

1. 用户导入视频并开始处理。
2. ASR 正常生成原始分段文稿。
3. 若存在第一个可用 LLM 配置，则系统自动执行 AI 纠错。
4. AI 纠错成功后，前台静默显示纠错后的文稿。
5. 后续所有依赖 `transcripts` 的功能默认使用纠错后的文本。

### 无可用 LLM 配置

1. ASR 完成后直接保留原始文稿。
2. 整条流程仍然成功。
3. 界面提供轻量提示，说明配置模型后可自动纠正文稿。

### AI 纠错失败

1. 原始 ASR 文稿仍然已可用。
2. 本次 AI 纠错结果被丢弃，不污染正式文稿。
3. 整条流程仍然成功，不要求用户立即重试。

## 数据设计

### 现有表的角色

`transcripts` 继续作为“当前正式文稿”表，不改变现有读路径。前端文稿面板、AI 生成链路、RAG、搜索和导出继续只读这张表。

### 新增备份表

新增一张内部备份表，建议命名为 `transcript_backups`：

- `id`
- `video_id`
- `source`：当前只使用 `raw_asr`
- `segments_json`：保存原始 ASR 分段快照
- `created_at`

`segments_json` 使用整包快照而不是逐行复制表，原因是：

- 这份数据不直接给前台读
- 主要用途是留档、回退和未来重跑
- 改动比复制第二套逐行表更小
- 不会干扰当前所有基于 `transcripts` 的查询和接口

建议快照结构包含每段的：

- `segment_idx`
- `start_ms`
- `end_ms`
- `text`
- `words_json`（如果原始 ASR 已提供）

## 处理流程

### 主流程

`audio -> asr -> 备份 raw_asr -> 写入原始正式文稿 -> 尝试 AI 纠错 -> 成功则覆盖 transcripts`

详细步骤如下：

1. ASR 返回分段结果。
2. 先把原始分段快照写入 `transcript_backups(source=raw_asr)`。
3. 再把原始分段写入 `transcripts`，保证即使没有 LLM 或后续纠错失败，文稿也立刻可用。
4. 检查是否存在第一个可用 LLM 配置。
5. 若没有可用配置，则结束在原始文稿。
6. 若存在可用配置，则对当前分段执行 AI 纠错。
7. 若 AI 纠错成功，则按分段覆盖 `transcripts.text`。
8. 若 AI 纠错失败，则保留原始 `transcripts` 内容并结束。

### 为什么不新增顶层阶段

本次不把 AI 纠错单独做成新的显式顶层阶段，而是仍然归在 `ASR` 阶段内部。这样可以：

- 避免立刻重做现有的处理阶段 UI
- 让用户把它理解成“更完整的转写”
- 保持实现集中在现有 ASR 链路，而不是拆散到多个消费方

阶段内的进度文案可以更细，例如：

- `准备识别引擎`
- `解析识别结果`
- `写入原始文稿`
- `正在 AI 纠正文稿`
- `完成`

## LLM 路由与触发条件

### 触发条件

只有在“已配置可用大模型”时才自动执行 AI 纠错。

### 路由策略

AI 纠错默认复用“第一个可用 LLM 配置”，不新增单独的“转写纠错”模型配置入口。

这样做的原因：

- 与当前 LLM 设置结构兼容，改动最小
- 用户无需为了这一个功能额外学习一套设置
- 先把主线打通，后续若需要再扩展为独立任务路由

## AI 纠错协议

### 纠错目标

AI 纠错被定义为“轻清洗”，不是总结或自由改写。它只应：

- 修正明显的识别错误
- 调整病句、断句、标点
- 清理部分口语赘词
- 尽量贴近原话，不补充视频中未说出的新知识

### 分段约束

- 尽量保持原有分段顺序
- 尽量保持原有时间戳边界
- 每段只返回纠正后的文本，不改写时间戳语义

### 输入

后端向模型提交按批次组织的分段数组，每段包含：

- `start_ms`
- `end_ms`
- `text`

### 输出

为降低 token 成本，模型输出只返回必要字段：

```json
[
  { "start_ms": 0, "end_ms": 3200, "text": "纠正后的文本" },
  { "start_ms": 3200, "end_ms": 6800, "text": "纠正后的文本" }
]
```

不要求返回原始文本、额外说明、解释性字段或自然语言总结。

### 批处理

长视频不应整篇一次性纠错。后端应按固定批次送模型，例如每批 30 到 80 段，具体批大小可按字符数或段数调优。批处理原则：

- 每批独立校验
- 按原顺序回写
- 不跨批重排时间戳

## 后端校验与回退

AI 返回结果在覆盖 `transcripts` 前必须通过结构校验。至少包括：

1. 返回值是合法 JSON 数组
2. 每项都包含 `start_ms`、`end_ms`、`text`
3. 时间戳能与当前批次的原始分段对上
4. 文本不是空字符串
5. 段顺序没有明显错乱

以下情况直接判定本次 AI 纠错失败：

- JSON 不合法
- 段数严重不匹配
- 时间戳无法对齐
- 大量空文本
- 模型返回明显解释性废话而非结构化结果

失败后处理：

- 丢弃本次 AI 纠错结果
- 保留原始 `transcripts`
- 记录日志，便于后续排查
- 不把整条处理流程标记为失败

## 对现有功能的影响

该方案的核心收益是：一旦 `transcripts` 被静默替换为纠错稿，现有依赖方无需单独改造即可获得更干净的输入，包括：

- 文稿面板
- AI 笔记
- AI 章节 / AI看
- 总结
- 测验
- 思维导图
- RAG 问答
- 文稿搜索
- SRT / VTT 导出

这意味着本次实现主要改变的是“正式文稿的生成质量”，不是每个消费功能的独立逻辑。

## 实现范围

本次实现预计会涉及以下区域：

- `src-tauri/migrations/`：新增 `transcript_backups` 表
- `src-tauri/src/pipeline/mod.rs`：把 AI 纠错挂进 ASR 主流程
- `src-tauri/src/pipeline/asr.rs`：补充原始快照存储与正式文稿覆盖辅助函数
- `src-tauri/src/pipeline/ai.rs` 或相邻模块：新增 transcript correction 调用与结果解析
- `src-tauri/src/llm/prompts.rs`：新增 transcript correction prompt
- 前端状态提示位置：仅保留轻量“可升级为自动纠错”的提示，不改整体交互结构

## 测试策略

### 单元测试

- prompt 中明确约束“只返回时间戳和纠正后文本”
- 解析器能处理合法 JSON 输出
- 非法 JSON、错乱时间戳、空文本能被拒绝

### 流程测试

- 有可用 LLM 时：ASR 后成功覆盖 `transcripts`
- LLM 调用失败时：流程仍成功且保留原始 `transcripts`
- 无 LLM 配置时：跳过纠错并保留原始 `transcripts`

### 回归关注点

- `transcript_text()` 拼装结果仍能正常供 notes / chapters / rag 使用
- 导出字幕仍读到可用文本
- 文稿面板轮询与编辑能力不受影响

## 风险与取舍

### 风险

- 轻清洗 prompt 若写得不够克制，模型可能过度改写原文
- 批次过大时，成本和失败率会升高
- 若只存一份 `raw_asr` 快照，后续想做更细粒度 revision history 还要继续扩展

### 当前取舍

- 先要“静默可用”的主线，而不是复杂的版本管理界面
- 先复用第一个可用 LLM 配置，而不是增加新的设置面
- 先把 AI 纠错纳入 ASR 阶段内部，而不是单独暴露为新处理阶段

## 后续扩展

完成本设计后，可在此基础上继续推进：

1. 一键生成讲义版 PDF，默认使用纠错后的正式文稿
2. 截图提问，优先把截图对应时间段的纠错稿作为上下文
3. 若未来有需要，再扩展原始稿 / 纠错稿对照视图或 revision history
